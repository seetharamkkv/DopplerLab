# Simulated speed estimation (DopplerSim batches)

Estimate **source speed** (and optionally **trajectory path** and **closest-approach distance**) from **synthetic pass-by audio** generated by [DopplerSim 2.0](https://github.com/rohitharumugams/dopplersim_2.0). Labels come from the simulator and are encoded in clip filenames and per-sample metadata under each batch folder.

This track is **notebook-first** today: training and evaluation live in Jupyter notebooks rather than an installable CLI package (unlike `IDMT_experiments/` and `length_estimation/`).

**Question:** Can a model recover physically meaningful kinematics from Doppler-shifted spectrograms when geometry is known at generation time?

---

## Tasks (simulated benchmarks)

| Task | Type | Labels (typical) |
|------|------|------------------|
| **Source speed** | Regression | 10–50 m/s (primary focus of the main notebook) |
| **Trajectory path** | 3-class classification | `straight`, `parabola`, `bezier` (optional auxiliary head) |
| **CPA distance** | Regression | 5–120 m or wider ranges depending on batch config (optional auxiliary head) |

Clip names follow DopplerSim batch export conventions, for example:

```
{vehicle}_{path}_{speed}mps_{dist}m_{id}.wav
```

Each sample folder under `audio_clips/sample_*/` may also include precomputed **log-CQT** arrays (`Common/cqt.npy`), velocity scalars, and metadata JSON used by the training notebook.

## Dataset (required)

Audio and features are **not** stored in this repository. Generate batches with [DopplerSim 2.0](https://github.com/rohitharumugams/dopplersim_2.0) (Batch Generation tab) or use an existing export.

| Item | Value |
|------|-------|
| Default batch root | `data/batch_outputs/` (symlink or copy from simulator output) |
| Manifest | `<batch_name>/dataset.csv` |
| Clips | `<batch_name>/audio_clips/sample_*/` |
| Features | Precomputed CQT `.npy` per sample (when exported by the simulator) |

Point the notebook at your batch with environment variables or the **RUN CONTROL** cell:

```bash
export DOPPLER_BATCH_OUTPUTS=/path/to/static/batch_outputs
export DOPPLER_BATCH_NAME=model_test_1000   # smoke test batch name
```

For attaching a large batch disk on a cloud GPU VM, see [`ref_docs/instructions.md`](ref_docs/instructions.md).

## Setup (local or VM)

```bash
cd speed_estimation   # or repo root; notebook paths are repo-relative
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

pip install -U pip
pip install "jax[cuda12]" flax optax tqdm scikit-learn pandas matplotlib seaborn msgpack jupyter papermill
```

GPU is recommended for full training. CPU works for tiny smoke tests (`TEST = True` in the notebook).

Symlink simulator output into the repo (optional but convenient):

```bash
mkdir -p data
ln -sf /path/to/DopplerSim/static/batch_outputs data/batch_outputs
```

Verify JAX sees a GPU:

```bash
python -c "import jax; print(jax.devices())"
```

## Quick start

1. Open [`DopplerNet_SA_XPINN.ipynb`](DopplerNet_SA_XPINN.ipynb).
2. Edit the **RUN CONTROL** cell at the top (`DATASET_BATCH_NAME`, `EPOCHS`, resume flags).
3. Run all cells, or headless via papermill:

```bash
cd /path/to/DopplerLab
source .venv/bin/activate
export DOPPLER_BATCH_OUTPUTS=data/batch_outputs
export DOPPLER_BATCH_NAME=model_test_1000

papermill speed_estimation/DopplerNet_SA_XPINN.ipynb outputs/run.ipynb --log-output
```

**Smoke test:** set `TEST = True` in RUN CONTROL (few clips, one epoch).

**Resume:** set `RESUME_TRAINING = True` after the first epoch; checkpoints live under `experiments/test_dopplernet_sa_xpinn/checkpoints/`.

---

## Split policy

The main notebook builds **leakage-safe** train/val/test splits:

- One **scene** (clip) appears in only one split.
- Stratification within **speed bins** (~70/15/15) so each split sees the full speed range.

Cached split indices are stored under `experiments/.../setup_cache/` for reproducible resume.
